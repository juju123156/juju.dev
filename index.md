**1. 외부 AI API 기반 위험도 분석 기능 설계 및 개발**

**[Challenge]**

외부 AI API를 활용한 작업 위험도 분석 기능을 도입하며, 분석 결과가 나오기 전까지 작업을 등록할 수 없게 해달라는 현업 부서의 요구사항이 있었습니다.

**[Risk Analysis]**

일 평균 2만 건이 처리되는 환경에서 외부 시스템의 지연이나 장애가 발생할 경우, 수천 명의 현장 작업자가 대기하게 되는 심각한 운영 리스크를 포착했습니다. 이는 단순한 기술적 오류를 넘어 비즈니스 가용성에 치명적일 수 있다고 판단했습니다.

**[Decision & Action]**

- 역제안을 통한 정책 변경: 현업 부서에 '분석 장애 시 운영 마비 시나리오'를 공유하며, 분석 결과를 **'필수 조건'이 아닌 '가이드성 정보'**로 재정의할 것을 제안
- 낙관적 등록 후 사후 검증: 장애 상황에서도 우선 작업을 등록하되, 분석 결과가 위험으로 판명될 경우에만 관리자에게 즉시 알림을 보내는 사후 검증 모델 설계

**[기술적 격리]**

- 외부 AI API 연동을 위한 전체 데이터 흐름 설계 및 요청/응답 원본과 분석 결과를 분리한 기본 테이블 + 1:N 상세 구조의 데이터 모델 설계
- 기존 작업 등록 로직과의 결합도를 최소화하기 위해 **작업 ID 사전 생성 + 분석 데이터 선저장** 구조 적용
- 외부 API 장애가 핵심 트랜잭션에 영향을 주지 않도록 **분석 로직을 별도 트랜잭션으로 분리**하여 서비스 안정성 확보
- 외부 시스템 latency로 인한 응답 지연 방지를 위해 retry 제거 및 **timeout(5초) 기반의 fail-fast 구조** 적용

**[Result]**

> 실제로 발생한 외부 서버 장애 상황에서도 **작업 등록 프로세스 영향도 0% 달성**

**2. MySQL MMM 기반 Replication 장애로 인한 서비스 접속 불가 문제 해결**

**[문제]**

- 네트워크 지연으로 DB#1, DB#2 간 Replication이 Broken 상태 발생
- Monitor 서버(mmm-monitor) 비정상으로 DB 상태 판단 및 VIP failover 미동작
- Writer VIP가 정상적으로 할당되지 않아 모바일 로그인 등 서비스 접속 불가 발생

**[개선]**

- Replication 상태 및 Monitor 서버 이상 여부를 점검하여 장애 원인 식별
- Monitor 서버 재기동을 통해 상태 판단 기능을 복구하고 VIP 재할당 수행
- Primary DB를 기준으로 Replication 재정렬하여 데이터 정합성 및 서비스 정상화

**[성과]**

- 서비스 접속 불가 장애를 신속히 복구하여 정상 트래픽 처리 재개
- 장애 원인(Retention 지연 + Monitor 의존 구조)을 명확히 식별하여 대응 체계 정립

**[사후조치]**

- mmm-agent 및 mmm-monitor 프로세스에 대한 헬스체크 및 감시 로직 추가
- Monitor 서버 단일 의존 구조(SPOF)로 인한 장애 확산 가능성을 인지하고, Proxy 기반 DB 접근 구조 및 Failover 아키텍처 개선 방향에 대해 팀 내 논의 수행

**3. Kubernetes CronJob 기반 월 단위 배치 실행 구조 설계 및 구축**

- DB 조회 → 데이터 가공 → Excel 생성 → 게시판 업로드까지 이어지는 **end-to-end 데이터 파이프라인** 구성
- 파일 저장을 위해 **PV/PVC 및 S3 연동 구조** 설계하여 컨테이너 환경에서도 안정적인 파일 처리 가능하도록 개선
- 배치 실패 시 최대 3회 retry 및 실행 이력 DB 저장 구조를 설계하여 **중복 실행 방지 및 상태 기반 재처리** 가능하도록 구현
- 게시판 시스템과 연동하여 파일 업로드 및 메타데이터(DB) 정합성 유지 로직 구현
- **Sealed Secret**을 활용한 배치 환경의 민감 정보 관리 체계 적용

**1. AWS RDS 전환에 따른 DB 접근 구조 개선**

- RDS 인스턴스 분리로 기존 cross database query 구조가 동작하지 않는 문제 발생
- 공통팀과 협업하여 공통 DB 의존 데이터 범위를 분석
- 필요한 테이블을 **ETL 방식으로 업무 DB에 복제**하는 구조 설계
- Spring 환경에서 별도의 DataSource 및 SqlSessionTemplate을 구성하여 DB 접근 로직 분리

---

**2. Oracle → MariaDB SQL 마이그레이션 및 성능 개선**

- 작업 조회 쿼리 전환 과정에서 CTE 기반 대용량 JOIN 쿼리 성능 저하 문제 발생
- **Execution Plan(rows vs r_rows) 분석**을 통해 Nested Loop JOIN 반복으로 약 300만 회 인덱스 접근 발생하는 병목 구간 확인
- MariaDB 환경에서 Materialized View 미지원 제약을 고려하여 **Summary Table 구조** 설계
- 조회용 Summary Table 기반으로 전환하여 대량 JOIN을 제거하고 실행계획 안정화

**[성과]**

| 지표 | Before | After | 개선율 |
|------|--------|-------|--------|
| 쿼리 실행시간 | 16.47초 | 0.057초 | **99.7% 개선** |
| 인덱스 접근 횟수 | 약 300만 회 | ~0 | **거의 제거** |

---

**3. Batch 시스템 클라우드 전환 및 Observability 개선**

- 사내 Batch Scheduler 기반 Java Batch 작업을 **Kubernetes CronJob 기반 실행 구조**로 전환
- Java Batch 애플리케이션을 컨테이너화하고 **Helm Chart 기반 배포** 구성
- 애플리케이션 코드에 **Datadog SDK 연동**을 적용하여 배치 실행 로그 및 메트릭 수집
- Datadog 대시보드를 통해 배치 실행 상태 및 오류 발생 여부 모니터링 가능하도록 개선

**[성과]**

- 배치 실행 로그 및 메트릭이 Datadog으로 중앙 수집되어 **운영 모니터링 가시성 향상**
