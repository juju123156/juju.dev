# SK텔레콤 TANGO-O 개발

하루 2만 건 규모의 통신 작업을 안정적으로 처리할 수 있도록 시스템 구조를 개선하고 반복되는 수작업을 기술적으로 해소하는 데 집중했습니다.
다수 외부 시스템과의 연동 과정에서 발생하는 리스크와 휴먼 에러를 최소화하는 한편, 운영 효율을 극대화하기 위해 자동화, 데이터 정합성 검증, 인터페이스 개선 등에 신경썼습니다.

외부 연동 작업 실패율을 낮추기 위해 Retry(Exponential Backoff)·Circuit Breaker 기반 공통 연동 모듈을 구축하고,
요청/응답 Schema Validation 및 작업 단위 Idempotency Key 적용을 통해 데이터 정합성과 중복 처리 문제를 해결했습니다.
Batch 기반 자동화와 Structured Logging을 도입해 장애 탐지 시간과 MTTR을 단축했으며,
작업 처리 지표를 모니터링해 운영 효율을 개선했습니다.

**Stack**  
Java, Spring Boot, Spring MVC

# CV-API 기능
**작업자가 입력한 네트워크·통신 장비 명령어를 외부 AI API로 분석해, 서비스 중단 가능성이 있는 위험 명령어를 사전에 자동 감지하는 기능을 설계·개발**

- 기존에는 작업 등록시 위험도가 있는 명령어를 통신장비에 내리게 되면 장애를 미리 감지할 수 있는 지표가 부족하여 사용자가 리스크를 인지하기 어려웠습니다.
- 위험 작업을 사전에 식별·차단함으로써 안전하게 작업을 진행할 수 있게 되었고, 운영팀은 사전 검증 기반의 안정적 품질 관리를 가능하게 되었습니다.

- 외부 AI API 연동을 위한 전체 데이터 흐름을 설계하고, 요청/응답 원본과 분석 결과를 분리한 기본 테이블 + 1:N 상세 구조의 데이터 모델을 설계했습니다.
- 기존 작업 등록 로직과의 결합도를 최소화하기 위해 작업 ID를 사전 생성하고 분석 데이터를 선저장하는 구조로 설계했습니다.
- 외부 API 장애가 핵심 작업 등록 트랜잭션에 영향을 주지 않도록 분석 로직을 별도 트랜잭션으로 분리하여 서비스 안정성을 확보했습니다.
- 외부 시스템 latency로 인한 응답 지연이 발생하지 않도록 retry를 제거하고 timeout(5초) 기반의 fail-fast 구조를 적용했습니다.

# 외부작업 연동 동시성 제어
**리포트 생성 등 업무 로직 수행 시 여러 외부 API와 통신을 수행하며, 안정적 데이터 처리와 성능 확보를 위한 연동 구조 설계 및 개선**

- 외부 API 기반 리포트 생성 로직에서 동시 요청 시 중복 호출로 성능 저하가 발생했습니다.
- PostgreSQL Advisory Lock 기반 분산락과 멱등성 처리로 리포트 생성 API의 동시성 문제를 해결했습니다.
- 외부 API 중복 호출: 5회 → 1회 (80% 감소) 전체 처리 시간: 138초 → 61초 (56% 단축)
- 캐시 스탬피드 방지 및 동시성 문제 근본 해결, 멱등성을 확보했습니다.

### 설계 개선
- 요청별 응답시간 편차: 82~137초 → 60.7~60.8초로 수렴
  → 단일 요청만 API 호출, 나머지는 대기 후 즉시 반환 구조로 개선했습니다.
- PostgreSQL Advisory Lock 도입 및 잘못된 재시도 제거로 단일 요청 API만 호출하도록 처리했습니다.

# MySQL MMM 기반 Replication 설계 개선
작업 정보의 생성·상태 관리 및 이력 추적을 담당하는 MySQL 기반 운영 DB를 대상으로 Replication 구조를 개선했습니다.

- 네트워크 지연으로 DB#1, DB#2 간 Replication이 Broken 상태가 발생하여 작업 관련 서비스 접속이 불가했습니다.
- 기존 MMM 기반 Replication 구조를 MySQL Orchestrator 기반 자동 Failover 아키텍처로 전환하여 장애 복구 속도·데이터 정합성·운영 안정성을 크게 향상했습니다.

### 설계 개선  
- 장애 원인(Replication retention 지연 및 MMM Monitor 서버 단일 의존 구조)을 분석하고, 이를 해소하기 위해 MySQL MHA 기반의 자동 Failover 구조로 전환하는 방향을 제안했습니다.
- MMM Monitor 서버의 SPOF 문제를 제거하기 위해 MHA의 분리된 Manager/Node 구조를 적용하고,
- Failover 시 Proxy 라우팅 전환이 지연되는 문제를 해결하기 위해 Proxy 기반 DB 접근 구조 및 자동 승격 방식 개선을 팀 내에서 설계했습니다.

# 1. AWS RDS 전환에 따른 DB 접근 구조 개선

- RDS 인스턴스 분리로 기존 cross database query 구조가 동작하지 않았습니다.
- RDS 분리 환경에서 cross DB query 의존성을 제거하고 ETL 기반 데이터 복제 구조를 설계하여 서비스 안정성과 확장성을 개선했습니다.

### 설계 및 개선
- 공통팀과 협업하여 공통 DB 의존 데이터 범위를 분석
- 필요한 테이블을 **ETL 방식으로 업무 DB에 복제**하는 구조 설계
- Spring 환경에서 별도의 DataSource 및 SqlSessionTemplate을 구성하여 DB 접근 로직 분리

# 2. Oracle → MariaDB SQL 마이그레이션 및 성능 개선

- 작업 조회 쿼리 전환 과정에서 CTE 기반 대용량 JOIN 쿼리 성능 저하 문제가 발생했습니다.
- Oracle → MariaDB 마이그레이션 과정에서 대용량 JOIN 쿼리를 Summary Table 구조로 개선하여 실행시간을 99.7% 단축

### 개선
- **Execution Plan(rows vs r_rows) 분석**을 통해 Nested Loop JOIN 반복으로 약 300만 회 인덱스 접근 발생하는 병목 구간을 확인했습니다.
- MariaDB 환경에서 Materialized View 미지원 제약을 고려하여 **Summary Table 구조** 설계했습니다.
- 조회용 Summary Table 기반으로 전환하여 대량 JOIN을 제거하고 쿼리 실행시간을 6.47초 -> 0.057초  **99.7% 개선했습니다.
